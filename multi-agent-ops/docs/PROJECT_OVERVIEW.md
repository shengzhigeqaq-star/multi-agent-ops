# 多Agent协同运营自动化系统 (MAO) — 项目概述

## 一、核心痛点

### 1.1 单一LLM的质量天花板
单个大模型在面对复杂运营任务时存在根本性缺陷：
- **无自校正能力**：模型输出后缺乏独立的验证机制，错误和遗漏只能靠人工发现
- **长链推理断裂**：超过3步的推理链容易产生幻觉累积，每一步的误差逐级放大
- **角色混淆**：策划、执行、审核三种不同思维模式混在同一上下文中，导致输出质量下降

### 1.2 企业运营的复合性需求
真实运营场景天然是多角色的：
| 场景 | 涉及角色 | 传统方式痛点 |
|------|---------|------------|
| 市场分析报告 | 分析师(搜集) + 撰稿人(撰写) + 编辑(审核) | 一人包办质量差，多人协作效率低 |
| 数据运营 | 数据工程师(提取) + 分析师(建模) + 业务(解读) | 跨角色沟通成本高 |
| 内容运营 | 策划(选题) + 编辑(撰写) + 主编(审核) | 流程冗长，反复修改 |

### 1.3 现有Agent框架的局限
- **LangChain/LlamaIndex**：侧重单Agent+工具调用，缺少真正的多角色协作
- **AutoGPT/BabyAGI**：单Agent自循环，缺乏批判性审视视角
- **CrewAI/AutoGen**：支持多Agent但缺少长链推理的质量保障机制

**MAO的解决路径**：将"策划-执行-审核-协调"四角色固化到系统中，通过Agent间消息总线实现多轮交互验证，用审核-修改循环保证输出质量。


## 二、核心逻辑流

### 2.1 长链推理架构

整个系统就是一条长链推理流水线，每个环节有独立的验证和回退机制：

```
                    ┌──────────────────────────────────────┐
                    │        长链推理流水线                  │
                    │                                      │
  用户输入 ──→ Planner ──→ Executor ──→ Reviewer ──→ Coordinator ──→ 最终输出
               (分解)       (执行)       (审核)       (综合)          │
                  │            │           │            │              │
                  │            │    ┌──────┘            │              │
                  │            │    │ REVISE (回退)     │              │
                  │            │    ▼                   │              │
                  │            └── Executor ──→ Reviewer │              │
                  │              (重新执行)    (再审)    │              │
                  │                          │          │              │
                  │                     PASS │          │              │
                  └──────────────────────────┴──────────┘              │
                                                                      │
                              最多可迭代 3 轮修改                        │
                    └──────────────────────────────────────┘
```

**推理深度**：单个子任务最多经历 `Plan → Execute → Review → Revise → Execute → Review → Pass` 共6步推理链。若有5个子任务，整个任务链可达30+步推理。

### 2.2 多Agent协作协议

```
┌──────────┐    TASK      ┌──────────┐    RESULT    ┌──────────┐
│  Planner │  ←──────────→│ Executor │  ←──────────→│ Reviewer │
│  (规划师) │              │ (执行者)  │              │ (审核员)  │
└──────────┘              └──────────┘              └──────────┘
      │                        │                        │
      │                        ▼                        │
      │              ┌─────────────────┐                │
      └──────────────→│  Coordinator   │←───────────────┘
                      │   (协调员)      │
                      └─────────────────┘
                              │
                        最终综合输出
```

**四种协作模式**：

| 模式 | 说明 | 触发条件 |
|------|------|---------|
| **顺序流水线** | Plan → Execute → Review → Synthesize | 默认模式 |
| **并行执行** | 无依赖子任务并行分配给多个Executor | DAG拓扑排序 |
| **审核回退** | Reviewer返回REVISE时，Executor重新执行 | 质量不达标 |
| **冲突仲裁** | 多个Agent意见不一致时，Coordinator裁决 | 评审争议 |

### 2.3 完整执行流程（以"市场分析报告"为例）

```
Step 1:  Orchestrator 接收任务描述
         "写一份关于AI Agent市场的分析报告"

Step 2:  Orchestrator → Planner
         "请分析并分解此任务"
         Planner 思考链:
         ① 理解任务目标：市场分析报告
         ② 识别子任务：市场调研 → 竞争分析 → 趋势判断 → 撰写 → 审核
         ③ 确定依赖关系：趋势判断依赖市场调研+竞争分析
         ④ 输出结构化计划JSON

Step 3:  Orchestrator → Coordinator
         "请根据计划调度执行"
         Coordinator 分析:
         ① 检查依赖关系：sub_1和sub_2无依赖，可并行
         ② 分配资源：sub_1 → Executor-1, sub_2 → Executor-2
         ③ 设定审核门禁：所有执行结果需通过Reviewer

Step 4:  [并行] Executor-1 执行 "搜索AI Agent市场数据"
         ① 调用 web_search("AI Agent market size 2025")
         ② 调用 web_fetch("gartner.com/ai-agents")
         ③ 调用 web_fetch("marketsandmarkets.com/ai-agents")
         ④ 汇总数据 → 输出市场数据摘要

         [并行] Executor-2 执行 "分析竞争格局"
         ① 调用 web_search("AI Agent platforms comparison")
         ② 提取主要玩家和差异化
         ③ 输出竞争分析表

Step 5:  Reviewer 审查 sub_1 结果
         ① 检查数据来源是否可靠
         ② 验证数据是否包含2025年最新数据
         ③ 结论：PASS (评分 8/10)

         Reviewer 审查 sub_2 结果
         ① 检查竞争分析是否全面
         ② 发现遗漏：缺少中国市场玩家
         ③ 结论：REVISE (评分 5/10)
         ④ 修改建议："请补充中国AI Agent市场参与者"

Step 6:  Executor-2 重新执行（长链推理关键环节）
         ① 收到修改意见
         ② 调用 web_search("中国AI Agent厂商")
         ③ 补充百度、阿里、字节等国内玩家
         ④ 重新输出完整竞争分析

Step 7:  Reviewer 重新审核 sub_2
         ① 检查修改是否充分
         ② 结论：PASS (评分 8/10)

Step 8:  Executor 执行 "撰写报告"（依赖sub_1和sub_2都通过）
         ① 整合市场数据 + 竞争分析
         ② 撰写结构化报告
         ③ 输出报告初稿

Step 9:  Reviewer 审核报告
         ① 结论：PASS (评分 8/10)
         ② 小建议：增加执行摘要

Step 10: Coordinator 综合所有结果
         ① 整合所有子任务输出
         ② 调和潜在矛盾
         ③ 生成最终交付物：完整的AI Agent市场分析报告

总推理步数: 10步 (含1次审核回退修正)
Token消耗: 取决于各Agent使用的模型和工具调用次数
```

### 2.4 长链推理的关键机制

**1. 状态持久化**
每个推理步的状态（思维链、工具调用、审核结果）都持久化到SQLite，支持断点恢复和完整审计。

**2. 上下文累积**
AgentContext在推理链中传递，每个Agent都能看到：
- 当前任务目标
- 历史推理步骤
- 共享内存中的关键发现
- 前序Agent的产出

**3. 审核门禁**
Reviewer是质量保障的关键环节：
- 对照成功标准逐项检查
- 给出可操作的修改建议（而非仅仅"不通过"）
- 多次不通过触发升级到Coordinator决策

**4. 向量记忆**
ChromaDB存储历史任务的经验片段，后续任务可通过语义搜索复用之前的推理成果。

### 2.5 与MIMO平台的契合点

| MIMO关注点 | MAO的体现 |
|-----------|---------|
| **Token消耗** | 多Agent多轮协作自然产生大量Token消耗，且每次消耗都有明确的业务价值 |
| **长链推理** | Plan→Execute→Review→Revise→Re-execute 形成高质量长推理链 |
| **多Agent协作** | 4角色+消息总线+审核回退，展示真正的Agent协作 |
| **实用价值** | 面向运营自动化，可直接用于市场分析、数据报告、内容生产 |
| **可扩展性** | 插件式LLM后端+可注册工具+自定义工作流模板 |
